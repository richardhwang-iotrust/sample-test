가장 먼저 읽히는 핵심 지시 파일

# 프로젝트 작업 규칙
> 이 프로젝트는 Claude Code 하네스 보일러플레이트입니다.
## 세션 시작 시
새 대화가 시작되면 반드시 /start 를 실행한다.
## 작업 흐름 (순서 엄수)
1. 기능 구현
2. 로컬 확인 (사용자 확인 요청)
3. 커밋 & 푸시
## 큰 기능 요청 시
planner → designer → frontend → backend
## 세부 규칙 참조
.claude/rules/ git.md · communication.md · security.md

자동으로 먼저 읽히는 종이 + 다른 규칙 파일을 함께 데려오는 목차 역할

6요소 · 06 · CLAUDE.md

CLAUDE.md, 왜 핵심인가

3가지 특징

01
자동으로 읽힘
새 대화를 열 때마다 Claude가 가장 먼저 보는 종이. 따로 명령 없이 항상 로드.
02
규칙의 출발점
여기서 rules/ 안 자세한 규칙들을 불러옴. 모든 규칙은 이 파일에서 시작.
03
경제적 — 토큰 절약
매번 대화로 설명하던 걸 한 번에 — 단 너무 길면 오히려 성능 저하. 약 200줄 이내 유지, 세부 규칙은 rules/ 로 분리.
주의 CLAUDE.md는 권고 사항이라 Claude가 무시할 수 있어요 — 반드시 지키게 하려면 .claude/hooks/로 강제하세요

rules — 3가지 규칙집

Claude가 일관되게 행동하도록 정해 둔 규칙 모음

소통 방식
communication.md
  • 에러 → 쉬운 말로 풀이
  • 영향 범위 먼저 알림
  • 요청 애매하면 질문 먼저
  • 완료 시 결과 요약
저장 · 배포
git.md
  • 변경 기록(커밋) 메시지 형식 통일
  • 작업 폴더(브랜치) 이름 규칙
  • 최종본(main)에 직접 올리기 금지
  • 리뷰 후 깔끔하게 합치기
비밀값 · 인증
security.md
  • 비밀키 값(예: 구글 로그인 키)은 비밀 보관 파일에 직접
  • 안전한 로그인 방식 권장
  • 오래된 보안이 약한 토큰은 새로 만들지 않기
  • 비밀값이 노출되면 즉시 비활성화

가볍게 시작, 자주 수정

하네스는 한 번에 완성되지 않습니다

느슨하게 시작
→
써본다
→
불편한 부분 발견
→
한 줄 고친다

좋은 하네스는 점점 단순해진다 — 복잡해지고 있다면 뭔가 잘못된 것

✅ 잘 가고 있는 신호
  • 같은 지적을 두 번 안 해도 됨
  • 실수가 규칙이 된다 (자동 축적)
  • 차단 장치가 뭔가를 막고 있다
  • 불필요한 규칙이 줄어든다
⚠ 막힌 신호
  • 확인에 시간이 더 오래 걸린다
  • 시켰는데 원하는 결과가 안 나온다
  • 규칙·도구가 많은데 잘 안 쓴다
  • 설명서가 길어지고 관리가 안 된다

FAQ · 10

자주 묻는 질문

비개발자가 자주 묻는 4가지

Q. 기존 프로젝트에도 적용할 수 있나요?
네, 새 프로젝트가 아니어도 됩니다. 이 레포의 .claude/ 폴더와 CLAUDE.md 파일을 그대로 복사해 넣으세요. 그 다음 VS CODE · CLAUDE 채팅창에 /plugin install harness-session@harness-local 두 줄만 한 번씩 입력하면 끝나요.
Q. 한 번 세팅하면 팀원 모두에게 자동 적용?
코드처럼 git 으로 push 하면 팀 전체에 반영됩니다. 단, 플러그인은 각자 한 번씩 설치해야 해요. VS CODE · CLAUDE 채팅창에서 /plugin install ... 명령을 한 번 실행하면 그 PC 에선 계속 동작합니다.
Q. 플러그인이 탐색기에 왜 안 보여요?
플러그인은 프로젝트 폴더가 아니라 사용자 홈(~/.claude/)에 캐시 됩니다. 그래서 탐색기에 파일이 없어 보여도 정상이에요. VS CODE · CLAUDE 채팅창에서 /plugin install <이름> 한 번 실행 → 이후엔 자동 인식됩니다.
Q. 규칙에 없는 내용은 어떻게 동작하나요?
Claude Code 가 평소처럼 알아서 답하고 실행합니다. 별도로 강제하는 규칙이 없으면 일반 동작 그대로예요. 자주 어긋나는 부분이 있으면 VS CODE · CLAUDE 채팅창에서 /wrap 으로 규칙화하면 됩니다.
