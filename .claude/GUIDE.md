# Claude 설정 가이드

> 이 문서는 비개발자 팀원을 위해 작성되었습니다.
> `.claude/` 폴더가 무엇인지보다, **어떤 마음으로 이걸 대하고 어떻게 내 것으로 만드는지**를 먼저 설명합니다.

---

## 한 줄 요약

`.claude/` 폴더는 **"Claude한테 보내는 업무 지침서"** 입니다.

---

## 핵심 마인드셋

Claude는 매번 새로운 대화를 시작할 때 아무것도 기억하지 못합니다.  
"저번에 말했잖아요"가 통하지 않아요.

그래서 이 폴더가 필요합니다.

> "내가 원하는 방식, 우리 팀의 규칙, 말하는 방식" 을  
> 파일로 써두면 Claude가 매번 대화를 시작할 때 자동으로 읽어옵니다.

직원 채용에 비유하면:
- 매번 같은 교육을 반복하는 대신
- 입사 첫날 건네주는 **업무 매뉴얼**을 만들어두는 것

**이 폴더는 그 매뉴얼입니다.**

---

## 내가 원하는 걸 반영하는 방법

### 방법 1. Claude한테 말로 요청하기 (가장 쉬운 방법)

Claude와 대화하다가 "앞으로는 이렇게 해줘"라고 말하면 됩니다.

예시:
- "앞으로 작업 완료하면 항상 어떤 파일을 수정했는지 목록으로 알려줘"
- "코드 설명할 때 비유를 꼭 써줘"
- "PR 만들 때 항상 나한테 먼저 확인하고 해줘"

그러면 Claude가 "이 내용을 rules 파일에 추가할까요?"라고 물어볼 거예요.  
"응"이라고 하면 자동으로 저장됩니다.

---

### 방법 2. 직접 파일 수정하기

말로 하기 어렵거나, 정확하게 남기고 싶을 때 직접 파일을 열어서 고칩니다.

| 고치고 싶은 것 | 파일 위치 |
|---|---|
| Claude가 나한테 말하는 방식 | `.claude/rules/communication.md` |
| 브랜치, 커밋, PR 관련 규칙 | `.claude/rules/git.md` |
| 절대 하면 안 되는 행동 | `.claude/hooks/guard.sh` |

파일을 열면 한글로 설명이 적혀 있으니 읽어보면서 수정할 수 있어요.  
모르는 부분은 Claude한테 "이 파일 보면서 설명해줘"라고 하면 됩니다.

---

## 각 파일이 하는 일

```
.claude/
├── rules/
│   ├── communication.md   ← Claude의 말투·소통 방식
│   └── git.md             ← 코드 저장·배포 규칙
├── commands/
│   └── start.md           ← /start 명령어 정의
├── hooks/
│   └── guard.sh           ← 실수 방지 잠금장치
└── settings.json          ← 잠금장치 연결 설정
```

### `rules/communication.md` — Claude의 말투 설명서

Claude가 어떻게 말해야 하는지 적혀 있습니다.

현재 설정:
- 기술 용어는 쉬운 말과 같이 씀 (예: "커밋(저장)")
- 뭔가 고치기 전에 어디에 영향 있는지 먼저 말함
- 에러 메시지를 그대로 복붙하지 않고 쉽게 풀어서 말함
- 요청이 애매하면 바로 작업하지 않고 먼저 질문함

→ 이 중 마음에 안 드는 게 있으면 해당 줄을 지우거나 수정하면 됩니다.

---

### `rules/git.md` — 코드 저장·배포 규칙

커밋, 브랜치, PR에 관한 규칙이 적혀 있습니다.

현재 설정:
- 브랜치 이름은 항상 `{내 이메일 앞부분}-기능명` 형식으로 만들기
  - 예) `sumin@iotrust.kr`로 접속 중이면 → `sumin-기능명`
  - Claude가 작업 시작 전에 `git config user.email`을 읽어서 자동으로 파악함
  - 팀원이 바뀌어도 파일 수정 없이 각자 이름으로 자동 적용됨
- main 브랜치에는 직접 올리지 않기 (반드시 PR 경유)
- 커밋 메시지 형식 통일 (feat:, fix: 등)
- `.env.local` 값이 바뀌면 1Password도 업데이트하라고 알려주기

→ 브랜치 규칙 자체를 바꾸고 싶을 때만 이 파일을 수정합니다.

---

### `commands/start.md` — `/start` 명령어

Claude에게 `/start` 라고 입력하면 실행되는 동작입니다.

현재 동작:
1. 지금 작업자가 누군지 확인
2. 현재 브랜치 확인 (main이면 새 브랜치 안내)
3. 최신 main 내용 가져오기
4. 저장 안 된 작업 있으면 알려주기
5. 준비 완료 요약 메시지

→ 매일 작업 시작 전에 `/start` 입력하는 습관을 들이면 좋습니다.

---

### `hooks/guard.sh` — 실수 방지 잠금장치

이건 Claude가 아니라 **시스템이 직접 막아주는 장치**입니다.  
아무리 Claude에게 요청해도 아래 행동은 실행이 안 됩니다:

| 막는 행동 | 이유 |
|---|---|
| main 브랜치에 직접 push | 검토 없이 배포되는 사고 방지 |
| `--no-verify` 옵션 사용 | 안전 검사 우회 금지 |
| 커밋 메시지에 `Co-Authored-By` 포함 | 팀 커밋 기록 정리 |

→ 여기서 막히면 방법이 잘못된 겁니다. 우회하려 하지 말고 올바른 방법을 확인하세요.

---

## 자주 있는 상황별 안내

### "Claude가 자꾸 이상하게 행동해요"

먼저 `rules/communication.md`를 열어보세요.  
그 파일에 적힌 규칙대로 행동하고 있을 가능성이 높습니다.  
규칙이 현실과 달라졌다면 파일을 수정하거나 Claude한테 "이 규칙 바꿔줘"라고 하면 됩니다.

---

### "새 팀원이 생겼어요"

`rules/git.md`의 브랜치 prefix 예시를 보고,  
새 팀원의 이메일 앞부분이 자동으로 prefix가 된다는 걸 알려주세요.  
별도로 파일을 수정할 필요는 없습니다.

---

### "이 규칙 없애고 싶어요"

해당 파일에서 그 내용을 삭제하면 됩니다.  
삭제 후 Claude와 새 대화를 시작하면 바로 반영됩니다.

---

### "이 설정이 적용됐는지 어떻게 알아요?"

Claude와 대화를 시작할 때 `/start` 를 입력해보세요.  
응답 방식, 말투, 동작 순서가 이 파일들에 적힌 대로라면 제대로 적용된 겁니다.

---

## 마지막으로

이 폴더의 규칙들은 **고정된 것이 아닙니다.**

Claude와 일하다 보면:
- 불편한 게 생기고
- 더 좋은 방식을 발견하고
- 팀 상황이 바뀌기도 합니다

그때마다 파일을 조금씩 고쳐가는 것, 그게 이 폴더를 잘 쓰는 방법입니다.  
Claude한테 "이 부분 고쳐줘"라고 하면 직접 수정해 주기도 합니다.

> **이 폴더는 한 번 만들고 끝내는 게 아니라, 같이 쓰면서 계속 다듬는 것입니다.**
