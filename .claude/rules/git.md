# Git 작업 규칙

## 작업자 정보

세션 시작 시 아래 명령으로 작업자 이름을 자동으로 확인한다:

```bash
git config user.email
```

- `@` 앞의 이름을 브랜치 prefix로 사용한다
- 예) `dev@iotrust.kr` → prefix는 `dev`
- 브랜치 생성 시 항상 이 prefix를 사용한다: `{prefix}-기능명`

---

## 브랜치 전략

### 작업 시작 전 공통 체크

어떤 작업이든 시작 전에 확인:

1. 현재 브랜치명이 지금 할 작업과 맞는가?
   - 맞으면: 작업 진행
   - 다르면: 현재 브랜치 마무리(머지) 후 적절한 브랜치로 이동

### "큰 작업"의 기준 (아래 중 하나라도 해당되면 큰 작업)
- 현재 브랜치명과 작업 방향성이 다른 경우
- 새 기능 추가
- 구조 변경 (폴더, 파일 이동/삭제 포함)
- 기존 흐름에 영향을 주는 변경

### Pull 타이밍 (충돌 방지)
- **작업 시작 전**: 반드시 `git pull origin main` 실행
- **매일 1~2회**: 장시간 작업 시 중간에 pull 권장
- **머지 전**: PR 생성 직전 최신 main 반영 확인

### 큰 작업 시작 전 순서
1. `git pull origin main` 으로 최신 상태 확인
2. 현재 브랜치에 미완료 작업이 있으면 **머지까지 완료한다**
3. 머지 완료 후 새 브랜치를 생성하고 작업을 시작한다

### main 브랜치 보호
- main에 직접 push 금지
- 모든 변경은 반드시 PR을 통해서만 반영

## 커밋 메시지 형식

```
type: 한글 설명

- 상세 내용 1
- 상세 내용 2
```

### type 종류
- `feat`: 새 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 작업
- `refactor`: 코드 구조 변경
- `chore`: 설정, 기타 작업

### 주의
- 커밋 메시지에 `Co-Authored-By:` 줄 절대 추가 금지

## 머지 전 체크리스트

1. 모든 변경 파일이 커밋되었는지 확인
2. 커밋한 내용이 원격 레포지토리에 푸시되었는지 확인
3. 최신 main 내용을 풀해서 반영했는지 확인
4. 현재 브랜치가 작업 브랜치인지 확인 (main에서 직접 머지 금지)
5. 브랜치명이 `{prefix}-기능명` 형식인지 확인

## 머지 명령

```bash
# PR 생성 + 즉시 머지 (squash merge)
gh pr create --title "..." --body "..." && gh pr merge --squash --auto
```

## PR 템플릿

```
📌 이슈 제목
[type] 작업 제목

📌 브랜치명
{prefix}-기능명

📌 이슈 내용
작업 배경 또는 목적 한 줄 요약

- 세부 작업 내용 1
- 세부 작업 내용 2
- 세부 작업 내용 3
```

## 환경 변수 변경 시 팀 공유 안내

`.env` 파일은 git이 추적하지 않으므로, 환경 변수나 토큰 값이 바뀌었을 때 작업자에게 팀 내 시크릿 관리 도구에도 업데이트하도록 안내한다.

> 팀에서 사용하는 시크릿 관리 방식을 여기에 기입한다. (예: 1Password, Vault, Doppler, Notion 보안 페이지 등)
