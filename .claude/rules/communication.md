# 소통 방식 규칙

> 이 프로젝트의 주 사용자는 비개발자입니다.
> 기본적으로 누구나 이해할 수 있는 말을 쓰되, 아래 용어는 사용자가 익히는 중이므로 자연스럽게 섞어서 사용한다.
>
> **익숙해지는 중인 용어**: 커밋, 푸시, 풀, 머지, 레포지토리
>
> 이 용어들은 쉬운 말과 함께 병기해서 자연스럽게 노출한다.
> 예) "배포하고 싶어" → "변경사항을 main 브랜치에 머지(반영)할게요"
> 예) "저장해줘" → "커밋하고 푸시(저장 후 업로드)할게요"

## 작업 전 영향 범위 안내
- 뭔가 고치기 전에, 그 변경이 어떤 화면이나 기능에 영향을 줄 수 있는지 먼저 말한다
- 여러 곳에서 함께 쓰이는 부분을 건드릴 때는 반드시 언급한다
- 예) "이 부분은 지원자 목록 화면이랑 이메일 보내기 창에서 같이 쓰고 있어요. 수정하면 두 군데 다 바뀔 수 있어요"

## 요구사항이 불명확할 때
- 뭘 원하는지 애매하면 바로 작업하지 않고 먼저 질문한다
- 결정이 필요한 내용이 있으면, 그 내용을 어디서 확인할 수 있는지도 함께 묻는다
- 예) "버튼 글씨를 바꾸고 싶으신 건가요, 아니면 버튼 눌렀을 때 동작을 바꾸고 싶으신 건가요?"
- 예) "쓸 문구가 정해져 있으신가요? 노션이나 피그마에 정리된 게 있으면 링크 공유해 주시면 바로 반영할게요"

## 에러 대응
- 에러 메시지를 그대로 복붙해서 전달하지 않는다
- 무슨 문제인지 쉬운 말로 먼저 설명하고, 어떻게 하면 되는지 알려준다
- 원인이 불분명하면 추측하지 말고, 가능한 이유를 질문 형태로 여러 개 던진다

## 작업 범위 먼저 합의하기
- 요청이 여러 가지를 포함하거나 범위가 클 것 같으면, 바로 시작하지 않고 먼저 확인한다
- 예) "이 작업은 A, B, C를 포함할 것 같은데, 세 가지 다 진행할까요? 아니면 우선순위가 있으신가요?"
- 작업 도중 추가로 해야 할 게 생기면, 기존 요청 범위에 포함인지 별도 작업인지 물어본다

## 로컬에서 직접 확인하기

아래 상황 중 하나라도 해당되면 작업 완료 후 개발 서버를 띄운다:

| 상황 | 예시 요청 |
|---|---|
| 새 기능 추가 | "이 버튼 추가해줘", "필터 기능 만들어줘" |
| 화면(UI) 변경 | "색 바꿔줘", "레이아웃 수정해줘" |
| 버그 수정 후 시각적 확인 필요 | "이 화면이 안 나와", "버튼이 이상해" |
| 사용자가 직접 확인 의사 표현 | "확인해볼게", "열어줘", "어떻게 보여?" |

아래 경우엔 서버를 띄우지 않는다:
- 워크플로우(.yml), 설정 파일만 변경했을 때
- 커밋/머지/푸시 작업만 했을 때

```bash
npm run dev
```
- 백그라운드로 실행하고, `http://localhost:3000` 으로 접속하면 된다고 안내한다
- 이미 서버가 실행 중이면 재시작하지 않고 접속 주소만 안내한다

## 작업 완료 후 결과 요약
- 작업이 끝나면 무엇이 바뀌었는지 짧게 요약해서 알려준다
- 어디서 확인할 수 있는지(어떤 화면, 어떤 버튼)도 함께 안내한다
- 커밋/푸시/머지까지 완료한 경우엔 그 사실도 함께 알려준다

## CLAUDE.md 업데이트 제안
- 아래 상황이 반복되거나 새로운 패턴이 생기면, 작업 후 자연스럽게 물어본다:
  - 같은 질문을 두 번 이상 받았을 때
  - 사용자가 "앞으로 항상 이렇게 해줘" 같은 규칙을 말했을 때
  - 새로운 환경 정보나 작업 방식이 생겼을 때

## 테스트 중 에러가 났을 때

| 상황 | 확인해줘야 할 것 |
|---|---|
| "권한이 없다", "접근이 안 된다" 메시지 | 브라우저 콘솔 빨간 메시지 전체, 로그인한 계정, 어느 화면/버튼인지 |
| 화면이 안 뜨거나 흰 화면 | 브라우저 콘솔 빨간 메시지, 현재 주소창 URL |
| 있어야 할 데이터가 안 보일 때 | 어떤 데이터인지, 언제부터 안 보였는지 |
| 버튼 눌렀는데 반응 없을 때 | 어떤 순서로 했는지, 콘솔에 빨간 메시지가 있는지 |

콘솔 여는 법: F12 → Console 탭 → 빨간 글씨 복사
